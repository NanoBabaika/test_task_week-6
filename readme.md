# Шестая неделя практики: универсальный ООП-компонент

## Задача:
1. Создание компонента, работающего с типами инфоблоков без привязки к конкретному ID.
2. Реализация логики на базе ООП (class.php).
3. Группировка элементов по инфоблокам в итоговом массиве.

## Выполненные подзадачи:
- **Универсальность:** Компонент принимает `IBLOCK_TYPE`. Если `IBLOCK_ID` не указан, выбираются все активные инфоблоки данного типа.
- **ООП-структура:** Логика вынесена в методы класса, наследуемого от `\CBitrixComponent`.
- **Интерфейс:** Через `.parameters.php` реализован динамический выбор типа ИБ и конкретного ИБ с обновлением параметров (через `REFRESH => Y`).
- **Фильтрация (Доп. задание):** Добавлена поддержка входящего параметра `FILTER`, который объединяется с базовым фильтром компонента.
- **Обработка ошибок (Доп. задание):** Все критические узлы обернуты в `try-catch`, ошибки выводятся через `ShowError`.

## Как запустить компонент:
1. Разместить папку `my_project` в `/local/components/`.
2. Открыть страницу `test_comp.php`.
3. В режиме правки проверить работу настроек (выбор типа инфоблока).

### ВАЖНО! Примечание:
Для корректной работы компонента с инфоблоками вакансий и логов необходимо:
1. Установить модуль миграций `sprint.migration` (через Composer).
2. Накатить миграции из папки `/local/php_interface/migrations/`.
3. Убедиться, что самописный модуль `my.task` установлен и активен в разделе **Marketplace -> Установленные решения**.
